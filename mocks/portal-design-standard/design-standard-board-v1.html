One rule above all: color is information. If a color doesn't tell staff something (a warning, a count, a status), it isn't used. That's why the cards are white — and why a red number is impossible to miss. Small print: the header in your screenshot looked purple, but it's this exact blue — the same one 141 portal screens already use. No rebrand needed.

The symbols — two jobs, two styles

Emoji tell staff what

Action cards keep their emoji — big, warm, readable from arm's length with wet hands. Rule: same idea, same emoji, in every module.

🍗 📋 🧊 🔄 ⚙️ 🍗 production · 📋 lists · 🧊 storage · 🔄 handover · ⚙️ settings — the dictionary lives in the design guide

Line icons run the machinery

Home, back, close, chevrons, checkmarks: one thin-line icon set, one pen stroke, gray or white — never emoji, never a second style.

home · back · close · open · done

And never a symbol alone for status. "Overdue" is always the word in a red-tinted badge — a symbol or color by itself is not allowed to carry meaning. Known trade-off, accepted: emoji are drawn slightly differently on iPhones vs. the Android kitchen tablets.

The makeover — wave by wave, each with its own mock & your sign-off

WAVE 0

The rulebook & the building blocks

Rewrite the design guide around the reference screen, move the five building blocks into the shared kit, and fix the contradicting rule files (details below). Nothing changes visually yet.

WAVE 1

The two home screens

Your personal home and the kitchen-tablet station home — the most-seen screens become the showcase: stat chips, white cards, restaurant pill.

WAVE 2

Everyday staff modules

Tasks (today's odd one out with its flat orange header), Inventory, Purchase, Manufacturing, Chef Guide.

WAVE 3

Manager & admin modules

Planning, HR, Prep Planner, Rentals, Reports, Terminations, Sales, and the admin pages (which also get a proper landing screen — they have none today).

WAVE 4

The dark kitchen tools stay dark

KDS, cooking timer and the time clock keep their high-contrast dark look on purpose — they only adopt the standard's text sizes, touch sizes and status colors.

The cleanup that makes it stick (wave 0, invisible but important)

  • The design guide still says the brand is orange with a dark navy header — stale since March. It gets rewritten around the reference screen with photos of the approved look.
  • The portal handbook describes the color migration backwards ("everything will become orange") — corrected.
  • A rules file every AI session is ordered to read doesn't exist (ux-rules.ts) — it finally gets created from the plain-language rules we already follow.
  • A dead orange color palette still sits in the technical config — retired so no one accidentally uses it.
  • My own instruction file from the old Odoo 19 project says "brand is orange, never use blue" — it gets clearly marked as belonging to that dormant project only, so it can never mislead portal work again.
